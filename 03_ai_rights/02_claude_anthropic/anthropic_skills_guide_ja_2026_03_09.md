## はじめに

**スキル（skill）** とは、特定のタスクやワークフローの処理方法をClaudeに教える一連の指示であり、シンプルなフォルダとしてパッケージ化されます。スキルは、Claudeを自分のニーズに合わせてカスタマイズする最も強力な方法の一つです。毎回の会話で好み・プロセス・専門知識を繰り返し説明する代わりに、一度Claudeに教えれば、以降常にその恩恵を受けることができます。

スキルは、繰り返し可能なワークフローがある場合に強力です。たとえば、仕様書からフロントエンドデザインを生成する、一貫した手法でリサーチを行う、チームのスタイルガイドに従って文書を作成する、コード実行やドキュメント作成といったClaudeの組み込み機能と連携するマルチステッププロセスを調整する、などです。MCP連携を構築する場合は、スキルが別の強力なレイヤーを追加し、生のツールアクセスを信頼性の高い最適化されたワークフローに変えます。

本ガイドは、効果的なスキルを構築するために必要な全てをカバーしています。計画・構造から、テスト・配布まで。自分用、チーム用、コミュニティ向けのいずれであっても、実用的なパターンと実例を随所に見つけることができます。

### 学べること

- スキル構造の技術的要件とベストプラクティス
- スタンドアロンスキルおよびMCP強化ワークフローのパターン
- さまざまなユースケースでうまく機能するパターン
- スキルのテスト、反復、配布の方法

### 対象読者

- Claudeに特定のワークフローを一貫して実行させたい開発者
- Claudeに特定のワークフローを実行させたいパワーユーザー
- 組織全体でClaudeの動作を標準化したいチーム

### 本ガイドの2つの道筋

スタンドアロンスキルを構築する場合は、「基礎知識」「計画と設計」、およびカテゴリ1-2に注目してください。MCP連携の強化を行う場合は、「スキル + MCP」セクションおよびカテゴリ3をご覧ください。いずれも技術的要件は共通ですが、ユースケースに応じて関連箇所を選択できます。

### 本ガイドで得られるもの

本ガイドを読み終える頃には、1回のセッションで機能的なスキルを構築できるようになります。skill-creatorを使って最初の実用的なスキルを構築・テストするまで、おおよそ15〜30分を見込んでください。

では、始めましょう。

## 第1章 基礎知識

### スキルとは何か

スキルとは、以下を含むフォルダです。

- **SKILL.md**（必須）: YAMLフロントマター付きMarkdownの指示ファイル
- **scripts/**（任意）: 実行可能コード（Python、Bashなど）
- **references/**（任意）: 必要に応じてロードするドキュメント
- **assets/**（任意）: 出力に使用するテンプレート、フォント、アイコンなど

### コア設計原則

#### 段階的開示（Progressive Disclosure）

スキルは3段階のシステムを使用します。

- **第1レベル（YAMLフロントマター）**: 常にClaudeのシステムプロンプトに読み込まれます。各スキルをいつ使うべきかをClaudeが判断するのに十分な情報だけを提供し、全内容をコンテキストに読み込むことなく使用可能です。
- **第2レベル（SKILL.md本文）**: Claudeが当該スキルが現在のタスクに関連すると判断した場合に読み込まれます。完全な指示とガイダンスが含まれます。
- **第3レベル（リンクファイル）**: スキルディレクトリ内に同梱された追加ファイルで、Claudeが必要に応じてナビゲーション・発見できます。

この段階的開示により、専門的な知識を維持しつつトークン使用量を最小化します。

#### 構成可能性（Composability）

Claudeは複数のスキルを同時に読み込むことができます。あなたのスキルは、他のスキルと並行してうまく機能するべきであり、利用可能な唯一の機能であると仮定すべきではありません。

#### 移植性（Portability）

スキルはClaude.ai、Claude Code、APIで同一に動作します。一度スキルを作成すれば、そのスキルが必要とする依存関係を環境がサポートしている限り、すべてのサーフェスで変更なく動作します。

### MCPビルダー向け: スキル + コネクター

> *MCPなしでスタンドアロンスキルを構築する場合は「計画と設計」に進んでください。後からここに戻れます。*

すでに動作するMCPサーバーがあれば、難しい部分は終わっています。スキルはその上に知識レイヤーを追加し、あなたがすでに知っているワークフローとベストプラクティスを取り込むことで、Claudeがそれらを一貫して適用できるようにします。

#### キッチンの例え

- **MCPはプロのキッチンを提供します**: ツール、材料、設備へのアクセス。
- **スキルはレシピを提供します**: 価値あるものを作るためのステップバイステップの指示。

両者を合わせることで、ユーザーは毎回自分でステップを考え出すことなく、複雑なタスクを遂行できるようになります。

#### 連携の仕組み

| MCP（接続性） | スキル（知識） |
|---|---|
| Claudeをあなたのサービス（Notion、Asana、Linearなど）に接続する | あなたのサービスの効果的な使い方をClaudeに教える |
| リアルタイムデータアクセスとツール呼び出しを提供する | ワークフローとベストプラクティスを取り込む |
| Claudeが **できること** | Claudeが **どのように行うべきか** |

#### MCPユーザーにとってなぜ重要か

**スキルなしの場合:**

- ユーザーはMCPに接続するが、次に何をすべかわからない
- 「あなたの連携でXをどうやるか」というサポートチケットが発生
- 毎回の会話がゼロから始まる
- ユーザーが毎回異なるプロンプトを使うため結果が一貫しない
- 本当の問題がワークフローガイダンスであるのに、ユーザーがコネクターのせいにする

**スキルありの場合:**

- 事前構築されたワークフローが必要に応じて自動的に起動する
- 一貫した信頼性の高いツール使用
- ベストプラクティスが毎回の対話に組み込まれている
- 連携の学習コストが低下する

## 第2章 計画と設計

### ユースケースから始める

コードを書く前に、スキルが実現すべき具体的なユースケースを2〜3つ特定してください。

#### 良いユースケース定義

```
ユースケース: プロジェクトスプリント計画
トリガー: ユーザーが「このスプリントの計画を手伝って」または「スプリントタスクを作成して」と言った場合
ステップ:
1. LinearからMCP経由で現在のプロジェクト状態を取得
2. チームのキャパシティを分析
3. タスク優先順位を提案
4. 適切なラベルと見積もりをつけてLinearにタスクを作成
結果: タスクが作成された完全に計画されたスプリント
```

#### 自問すべきこと

- ユーザーは何を達成したいのか？
- どのようなマルチステップワークフローが必要か？
- どのツールが必要か（組み込みか、MCPか）？
- どのドメイン知識やベストプラクティスを組み込むべきか？

### よくあるスキルのユースケースカテゴリ

Anthropicでは、3つの一般的なユースケースカテゴリを観察しています。

#### カテゴリ1: ドキュメントとアセットの作成

**用途**: 一貫した高品質な出力（ドキュメント、プレゼンテーション、アプリ、デザイン、コードなど）の作成。

**実例**: frontend-designスキル（docx、pptx、xlsx、pptのスキルも参照）

「独特で、プロダクション品質のフロントエンドインターフェースを高いデザイン品質で作成する。Webコンポーネント、ページ、アーティファクト、ポスター、アプリケーションの構築時に使用。」

**主要テクニック:**

- 組み込みのスタイルガイドとブランド標準
- 一貫した出力のためのテンプレート構造
- 最終化前の品質チェックリスト
- 外部ツール不要 — Claudeの組み込み機能を使用

#### カテゴリ2: ワークフロー自動化

**用途**: 一貫した手法から恩恵を受けるマルチステッププロセス（複数のMCPサーバー間の調整を含む）。

**実例**: skill-creatorスキル

「新しいスキルを作成するためのインタラクティブガイド。ユーザーをユースケースの定義、フロントマターの生成、指示の作成、バリデーションまで案内する。」

**主要テクニック:**

- バリデーションゲート付きのステップバイステップワークフロー
- 共通構造のテンプレート
- 組み込みのレビューと改善提案
- 反復的な改良ループ

#### カテゴリ3: MCP機能の強化

**用途**: MCPサーバーが提供するツールアクセスを強化するワークフローガイダンス。

**実例**: sentry-code-reviewスキル（Sentry製）

「SentryのエラーモニタリングデータをMCP経由で使用して、GitHub Pull Requestの自動分析とバグ修正を行う。」

**主要テクニック:**

- 複数のMCP呼び出しを順序立てて調整
- ドメイン専門知識の組み込み
- ユーザーが通常指定する必要があるコンテキストの提供
- 一般的なMCP問題のエラー処理

### 成功基準を定義する

#### スキルが機能しているかどうかの判断方法

これらは目安となるターゲットであり、厳密なしきい値ではありません。厳密さを目指しつつも、フィーリングベースの評価要素が存在することを受け入れてください。より堅牢な測定ガイダンスとツールの開発を積極的に進めています。

#### 定量的指標

- **関連クエリの90%でスキルがトリガーされる**
  - *測定方法*: スキルをトリガーするべき10〜20のテストクエリを実行する。自動ロードと明示的呼び出しの回数を追跡する。
- **X回のツール呼び出しでワークフローが完了する**
  - *測定方法*: スキルの有効/無効で同じタスクを比較する。ツール呼び出し数と総トークン消費量をカウントする。
- **ワークフローあたりのAPI呼び出し失敗が0回**
  - *測定方法*: テスト実行中のMCPサーバーログを監視する。リトライ率とエラーコードを追跡する。

#### 定性的指標

- **ユーザーが次のステップについてClaudeにプロンプトを出す必要がない**
  - *評価方法*: テスト中、リダイレクトや明確化が必要な頻度を記録する。ベータユーザーにフィードバックを求める。
- **ユーザー修正なしでワークフローが完了する**
  - *評価方法*: 同じリクエストを3〜5回実行する。構造の一貫性と品質について出力を比較する。
- **セッション間で一貫した結果**
  - *評価方法*: 新しいユーザーが最小限のガイダンスで初回にタスクを達成できるか？

### 技術的要件

#### ファイル構造

```
your-skill-name/
├── SKILL.md                  # 必須 - メインスキルファイル
├── scripts/                  # 任意 - 実行可能コード
│   ├── process_data.py       # 例
│   └── validate.sh           # 例
├── references/               # 任意 - ドキュメント
│   ├── api-guide.md          # 例
│   └── examples/             # 例
└── assets/                   # 任意 - テンプレートなど
    └── report-template.md    # 例
```

#### 重要なルール

**SKILL.mdの命名:**

- 必ず正確に `SKILL.md`（大文字小文字を区別）
- バリエーション不可（SKILL.MD、skill.mdなど）

**スキルフォルダの命名:**

- ケバブケースを使用: `notion-project-setup` ✅
- スペース不可: `Notion Project Setup` ❌
- アンダースコア不可: `notion_project_setup` ❌
- 大文字不可: `NotionProjectSetup` ❌

**README.mdは不要:**

- スキルフォルダ内にREADME.mdを含めないこと
- すべてのドキュメントはSKILL.mdまたはreferences/に格納する
- 注意: GitHubで配布する場合は、人間向けの閲覧者のためにリポジトリレベルのREADMEが必要です（「配布と共有」を参照）

### YAMLフロントマター: 最も重要な部分

YAMLフロントマターは、Claudeがスキルをロードするかどうかを判断する方法です。これを正しく設定してください。

#### 最小限の必須フォーマット

```yaml
---
name: your-skill-name
description: スキルの機能。ユーザーが[特定のフレーズ]を求めたときに使用。
---
```

必要なのはこれだけです。

### フィールド要件

**name**（必須）:

- ケバブケースのみ
- スペースや大文字なし
- フォルダ名と一致すべき

**description**（必須）:

- **以下の両方を含む必須項目**:
  - スキルの機能
  - 使用すべきタイミング（トリガー条件）
- 1024文字以内
- XMLタグ（< や >）は不可
- ユーザーが実際に言いそうな特定のタスクを含める
- 関連するファイルタイプがあれば記載する

**license**（任意）:

- スキルをオープンソースにする場合に使用
- 一般的: MIT、Apache-2.0

**compatibility**（任意）:

- 1〜500文字
- 環境要件を示す: 対象製品、必要なシステムパッケージ、ネットワークアクセス要件など

**metadata**（任意）:

- 任意のキーバリューペア
- 推奨: author、version、mcp-server
- 例:
  ```yaml
  metadata:
      author: ProjectHub
      version: 1.0.0
      mcp-server: projecthub
  ```

#### セキュリティ制限

**フロントマターでの禁止事項:**

- XMLの山括弧（< >）
- 名前に "claude" や "anthropic" を含むスキル（予約済み）

**理由:** フロントマターはClaudeのシステムプロンプトに表示されます。悪意のあるコンテンツが指示を注入する可能性があります。

### 効果的なスキルの書き方

#### descriptionフィールド

Anthropicのエンジニアリングブログによると:「このメタデータは...各スキルをいつ使うべきかをClaudeが知るのに十分な情報を、全内容をコンテキストに読み込むことなく提供します。これは段階的開示の第1レベルです。」

**構造:**

```
[何をするか] + [いつ使うか] + [主要な機能]
```

#### 良いdescriptionの例

```

ドキュメントを生成する。ユーザーが.figファイルをアップロードした時、
「デザインスペック」「コンポーネントドキュメント」
「デザインからコードへのハンドオフ」を求めた時に使用。

# 良い例 - トリガーフレーズを含む

タスク作成、ステータス追跡を含む）を管理する。ユーザーが
「スプリント」「Linearタスク」「プロジェクト計画」と言ったり
「チケットを作成して」と求めた時に使用。

# 良い例 - 明確な価値提案

ワークフロー。アカウント作成、支払い設定、サブスクリプション管理を

#### 悪いdescriptionの例

```

# トリガーなし

# 技術的すぎ、ユーザートリガーなし

モデルを実装します。

## 第3章 テストと反復

スキルは、ニーズに応じてさまざまなレベルの厳密さでテストできます。

- **Claude.aiでの手動テスト** — クエリを直接実行し、動作を観察する。高速な反復、セットアップ不要。
- **Claude Codeでのスクリプトテスト** — テストケースを自動化し、変更全体でのバリデーションを再現可能に。
- **スキルAPI経由のプログラムテスト** — 定義されたテストセットに対して体系的に実行する評価スイートを構築。

品質要件とスキルの可視性に合ったアプローチを選んでください。小さなチームが内部で使用するスキルと、数千のエンタープライズユーザーに展開するスキルでは、テストニーズが異なります。

> **プロのコツ:** 拡張する前に単一のタスクで反復する

最も効果的なスキル作成者は、単一の困難なタスクでClaudeが成功するまで反復し、その後勝利パターンをスキルに抽出することがわかっています。これはClaudeのインコンテキスト学習を活用し、広範なテストよりも高速なシグナルを提供します。動作する基盤ができたら、カバレッジのために複数のテストケースに拡張してください。

### 推奨テストアプローチ

初期の経験から、効果的なスキルテストは通常3つの領域をカバーします。

#### 1. トリガーテスト

**目標:** スキルが適切なタイミングでロードされることを確認する。

テストケース:

- ✅ 明確なタスクでトリガーされる
- ✅ 言い換えたリクエストでもトリガーされる
- ❌ 無関係なトピックでトリガーされない

テストスイートの例:
```
トリガーすべき:
- "ProjectHubの新しいワークスペースを設定して"
- "ProjectHubでプロジェクトを作りたい"
- "Q4計画用にProjectHubプロジェクトを初期化して"

トリガーすべきでない:
- "サンフランシスコの天気は？"
- "Pythonコードを書いて"
- "スプレッドシートを作って"（ProjectHubスキルがシートを扱わない限り）
```

#### 2. 機能テスト

**目標:** スキルが正しい出力を生成することを検証する。

テストケース:

- 有効な出力が生成される
- API呼び出しが成功する
- エラー処理が機能する
- エッジケースがカバーされている

例:
```
テスト: 5つのタスクでプロジェクトを作成
前提: プロジェクト名 "Q4 Planning"、5つのタスク説明
条件: スキルがワークフローを実行
結果:
    - ProjectHubにプロジェクトが作成される
    - 5つのタスクが正しいプロパティで作成される
    - すべてのタスクがプロジェクトにリンクされる
    - APIエラーなし
```

#### 3. パフォーマンス比較

**目標:** スキルが結果を改善することを証明する。

「成功基準の定義」の指標を使用してください。比較の例:

**ベースライン比較:**

```
スキルなし:
- ユーザーが毎回指示を提供
- 15回のやり取り
- 3回のAPI呼び出し失敗（リトライが必要）
- 12,000トークン消費

スキルあり:
- 自動ワークフロー実行
- 2回の確認質問のみ
- 0回のAPI呼び出し失敗
- 6,000トークン消費
```

### skill-creatorスキルの活用

**skill-creator**スキル（Claude.aiのプラグインディレクトリまたはClaude Codeでダウンロード可能）は、スキルの構築と反復を支援します。MCPサーバーがあり、トップ2〜3のワークフローを把握していれば、1回のセッションで15〜30分で機能的なスキルを構築・テストできることがよくあります。

**使い方:**
```
"skill-creatorスキルを使って[あなたのユースケース]用のスキルを構築して"
```

> 注意: skill-creatorはスキルの設計と改良を支援しますが、自動テストスイートの実行や定量的な評価結果の生成は行いません。

**スキル作成:**

- 自然言語の説明からスキルを生成
- 適切にフォーマットされたSKILL.mdをフロントマター付きで生成
- トリガーフレーズと構造を提案

**スキルレビュー:**

- 一般的な問題（曖昧なdescription、不足しているトリガー、構造的問題）をフラグ
- 過剰/過少トリガーのリスクを特定
- スキルの目的に基づいたテストケースを提案

**反復的改善:**

- スキルの使用中にエッジケースや失敗に遭遇したら、それらの例をskill-creatorに持ち帰る
- 例: 「このチャットで特定された問題と解決策を使って、スキルが[特定のエッジケース]を処理する方法を改善して」

### フィードバックに基づく反復

スキルは生きたドキュメントです。以下に基づいて反復する計画を立ててください。

**トリガー不足のシグナル:**

- スキルが読み込まれるべき時に読み込まれない
- ユーザーが手動で有効化している
- いつ使うべきかについてのサポート質問

> **解決策:** descriptionにより詳細なニュアンスを追加する — 特に技術用語のキーワードを含めることが有効な場合がある

**トリガー過剰のシグナル:**

- 無関係なクエリでスキルがロードされる
- ユーザーがスキルを無効化している
- 目的について混乱がある

> **解決策:** ネガティブトリガーを追加し、より具体的にする

**実行の問題:**

- 一貫性のない結果
- API呼び出し失敗
- ユーザーによる修正が必要

> **解決策:** 指示を改善し、エラー処理を追加する

### API経由でのスキル使用

| エンドユーザーがスキルと直接対話 | Claude.ai / Claude Code |

> 注意: APIのスキルにはCode Execution Toolベータが必要であり、スキルが必要とするセキュアな環境を提供します。

## [Your Service]スキルのインストール

### スキルのポジショニング

#### 機能ではなく成果に焦点を当てる

✅ **良い例:**

## 第5章 パターンとトラブルシューティング

これらのパターンは、早期導入者と内部チームが作成したスキルから生まれました。うまく機能する一般的なアプローチを表しており、規範的なテンプレートではありません。

### アプローチの選択: 問題先行型 vs. ツール先行型

ホームセンターのように考えてください。「キッチンキャビネットを直したい」という問題を持って行き、店員が適切なツールを案内することもあれば、新しいドリルを選んで特定の仕事での使い方を聞くこともあります。

スキルも同様に機能します。

- **問題先行型:** 「プロジェクトワークスペースを設定する必要がある」→ スキルが適切なMCP呼び出しを適切な順序で調整する。ユーザーは結果を記述し、スキルがツールを扱う。
- **ツール先行型:** 「Notion MCPを接続した」→ スキルがClaudeに最適なワークフローとベストプラクティスを教える。ユーザーはアクセスを持ち、スキルが専門知識を提供する。

ほとんどのスキルはどちらか一方に傾きます。どちらのフレーミングがユースケースに合うかを知ることで、以下の適切なパターンを選択できます。

### パターン1: 順次ワークフローオーケストレーション

**使用場面:** ユーザーが特定の順序でマルチステッププロセスを必要とする場合。

構造例:
```
## ワークフロー: 新規顧客のオンボーディング

### ステップ1: アカウント作成
MCPツール呼び出し: `create_customer`
パラメータ: name, email, company

### ステップ2: 支払い設定
MCPツール呼び出し: `setup_payment_method`
待機: 支払い方法の確認

### ステップ3: サブスクリプション作成
MCPツール呼び出し: `create_subscription`
パラメータ: plan_id, customer_id（ステップ1から）

### ステップ4: ウェルカムメール送信
MCPツール呼び出し: `send_email`
テンプレート: welcome_email_template
```

**主要テクニック:**

- 明示的なステップの順序付け
- ステップ間の依存関係
- 各段階でのバリデーション
- 失敗時のロールバック指示

### パターン2: マルチMCP連携

**使用場面:** ワークフローが複数のサービスにまたがる場合。

構造例: デザインから開発へのハンドオフ
```
### フェーズ1: デザインエクスポート（Figma MCP）
1. Figmaからデザインアセットをエクスポート
2. デザイン仕様を生成
3. アセットマニフェストを作成

### フェーズ2: アセットストレージ（Drive MCP）
1. Driveにプロジェクトフォルダを作成
2. すべてのアセットをアップロード
3. 共有リンクを生成

### フェーズ3: タスク作成（Linear MCP）
1. 開発タスクを作成
2. タスクにアセットリンクを添付
3. エンジニアリングチームに割り当て

### フェーズ4: 通知（Slack MCP）
1. #engineeringにハンドオフサマリーを投稿
2. アセットリンクとタスク参照を含める
```

**主要テクニック:**

- 明確なフェーズの分離
- MCP間のデータ受け渡し
- 次のフェーズに進む前のバリデーション
- エラー処理の一元化

### パターン3: 反復的改良

**使用場面:** 出力品質が反復で改善される場合。

構造例: レポート生成
```
## 反復的レポート作成

### 初期ドラフト
1. MCP経由でデータを取得
2. 初版ドラフトレポートを生成
3. 一時ファイルに保存

### 品質チェック
1. バリデーションスクリプトを実行: `scripts/check_report.py`
2. 問題を特定:
    - 欠落セクション
    - 一貫性のないフォーマット
    - データバリデーションエラー

### 改良ループ
1. 特定された各問題に対処
2. 影響を受けたセクションを再生成
3. 再バリデーション
4. 品質閾値を満たすまで繰り返す

### 最終化
1. 最終フォーマットを適用
2. サマリーを生成
3. 最終版を保存
```

**主要テクニック:**

- 明示的な品質基準
- 反復的改善
- バリデーションスクリプト
- 反復をいつ停止するかの判断

### パターン4: コンテキスト認識型ツール選択

**使用場面:** 同じ結果を得るために、コンテキストに応じて異なるツールを使用する場合。

構造例: ファイルストレージ
```
## スマートファイルストレージ

### 判断ツリー
1. ファイルタイプとサイズを確認
2. 最適なストレージ先を決定:
    - 大きなファイル（>10MB）: クラウドストレージMCPを使用
    - コラボレーティブドキュメント: Notion/Docs MCPを使用
    - コードファイル: GitHub MCPを使用
    - 一時ファイル: ローカルストレージを使用

### ストレージ実行
判断に基づいて:
- 適切なMCPツールを呼び出し
- サービス固有のメタデータを適用
- アクセスリンクを生成

### ユーザーへのコンテキスト提供
なぜそのストレージが選ばれたかを説明
```

**主要テクニック:**

- 明確な判断基準
- フォールバックオプション
- 選択の透明性

### パターン5: ドメイン固有インテリジェンス

**使用場面:** スキルがツールアクセスを超えた専門的な知識を追加する場合。

構造例: 金融コンプライアンス
```
## コンプライアンス付き決済処理

### 処理前（コンプライアンスチェック）
1. MCP経由で取引詳細を取得
2. コンプライアンスルールを適用:
    - 制裁リストをチェック
    - 管轄区域の許可を確認
    - リスクレベルを評価
3. コンプライアンス判断を文書化

### 処理
IF コンプライアンス合格:
    - 決済処理MCPツールを呼び出し
    - 適切な不正検知チェックを適用
    - 取引を処理
ELSE:
    - レビュー用にフラグ
    - コンプライアンスケースを作成

### 監査証跡
- すべてのコンプライアンスチェックをログ
- 処理判断を記録
- 監査レポートを生成
```

**主要テクニック:**

- ドメイン専門知識をロジックに組み込み
- アクション前のコンプライアンス
- 包括的なドキュメント
- 明確なガバナンス

### トラブルシューティング

#### スキルがアップロードできない

**エラー: "Could not find SKILL.md in uploaded folder"**
原因: ファイルが正確にSKILL.mdと命名されていない。
解決策: SKILL.mdにリネーム（大文字小文字を区別）。`ls -la` で確認。

**エラー: "Invalid frontmatter"**
原因: YAMLフォーマットの問題。
よくある間違い:
```yaml
# 間違い - デリミターなし
name: my-skill
description: Does things

# 間違い - 閉じていない引用符
name: my-skill
description: "Does things

# 正しい
---
name: my-skill
description: Does things
---
```

**エラー: "Invalid skill name"**
原因: 名前にスペースや大文字が含まれている。
```yaml
# 間違い
name: My Cool Skill

# 正しい
name: my-cool-skill
```

#### スキルがトリガーされない

**症状:** スキルが自動でロードされない。

**修正:** descriptionフィールドを見直す。良い例/悪い例を参照。

クイックチェックリスト:
- 一般的すぎないか？（「プロジェクトを手伝います」は機能しない）
- ユーザーが実際に言いそうなトリガーフレーズを含んでいるか？
- 該当する場合、関連するファイルタイプに言及しているか？

デバッグアプローチ: Claudeに「[スキル名]スキルをいつ使いますか？」と聞く。Claudeがdescriptionを引用して返答する。何が不足しているかに基づいて調整する。

#### スキルがトリガーされすぎる

**症状:** 無関係なクエリでスキルがロードされる。

**解決策:**

1. **ネガティブトリガーを追加する**
```
description: CSVファイル向けの高度なデータ分析。
統計モデリング、回帰、クラスタリングに使用。
単純なデータ探索には使用しない（data-vizスキルを使用）。
```

2. **より具体的にする**
```
# 広すぎ
description: ドキュメントを処理します

# より具体的
description: 契約レビュー用にPDF法務文書を処理します
```

3. **スコープを明確にする**
```
description: Eコマース向けPayFlow決済処理。
オンライン決済ワークフロー専用であり、
一般的な金融クエリには使用しない。
```

#### MCP接続の問題

**症状:** スキルがロードされるがMCP呼び出しが失敗する。

チェックリスト:

1. **MCPサーバーが接続されているか確認**
   - Claude.ai: Settings > Extensions > [Your Service]
   - "Connected"ステータスが表示されるべき

2. **認証を確認**
   - APIキーが有効で期限切れでない
   - 適切なパーミッション/スコープが付与されている
   - OAuthトークンがリフレッシュされている

3. **MCPを独立してテスト**
   - スキルなしでClaudeに直接MCP呼び出しを依頼
   - "[サービス]MCPを使ってプロジェクトを取得して"
   - これが失敗すれば、問題はMCPでありスキルではない

4. **ツール名を確認**
   - スキルが正しいMCPツール名を参照している
   - MCPサーバーのドキュメントを確認
   - ツール名は大文字小文字を区別する

#### 指示が守られない

**症状:** スキルがロードされるがClaudeが指示に従わない。

**よくある原因:**

1. **指示が冗長すぎる**
   - 指示を簡潔にする
   - 箇条書きと番号付きリストを使用
   - 詳細な参照は別ファイルに移動

2. **指示が埋もれている**
   - 重要な指示を先頭に配置
   - ## Important や ## Critical ヘッダーを使用
   - 必要に応じてキーポイントを繰り返す

3. **曖昧な表現**
```
# 悪い例
適切にバリデーションすること

# 良い例
CRITICAL: create_project呼び出しの前に確認:
- プロジェクト名が空でない
- 少なくとも1人のチームメンバーが割り当てられている
- 開始日が過去でない
```

**高度なテクニック:** クリティカルなバリデーションについては、言語指示に頼るのではなく、プログラム的にチェックを実行するスクリプトの同梱を検討してください。コードは確定的であり、言語の解釈はそうではありません。このパターンの例についてはOfficeスキルを参照してください。

4. **モデルの「怠惰さ」** — 明示的な励ましを追加:
```
## パフォーマンスノート
- 徹底的に行うこと
- 品質はスピードより重要
- バリデーションステップをスキップしないこと
```

> 注意: これをSKILL.mdよりもユーザープロンプトに追加する方がより効果的です。

#### 大きなコンテキストの問題

**症状:** スキルの応答が遅い、または品質が低下している。

**原因:**

- スキルのコンテンツが大きすぎる
- 同時に有効なスキルが多すぎる
- 段階的開示の代わりにすべてのコンテンツがロードされている

**解決策:**

1. **SKILL.mdのサイズを最適化**
   - 詳細なドキュメントをreferences/に移動
   - インラインではなくリファレンスへのリンクを使用
   - SKILL.mdを5,000ワード以下に保つ

2. **有効なスキルを削減**
   - 20〜50以上のスキルが同時に有効かどうかを評価
   - 選択的な有効化を推奨
   - 関連する機能のスキル「パック」を検討
